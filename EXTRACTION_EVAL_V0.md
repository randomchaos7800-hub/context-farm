# Extraction Eval v0

Status: draft  
Date: 2026-05-28

## Purpose

This document defines the first concrete extraction evaluation loop for Context Farm.

The point is to stop talking about extraction quality abstractly and start measuring it against a small labeled domain set.

## Scope

This first set is intentionally narrow:

- domain: `service-dispatch`
- labels: `Constraint`, `Exception`, `Procedure`, `Fact`, `Decision`, `Entity`, `Reject`
- examples: `16`

That is enough to evaluate:
- type classification
- title normalization
- review-priority assignment
- minimal statement coverage

## How To Use

1. Copy the prediction template.
2. Fill one prediction per gold item.
3. Run the evaluator.

Example:

```bash
cd /home/dino/context-farm-repo
cp examples/service-dispatch-extraction-predictions.template.json /tmp/preds.json
python3 scripts/eval_extraction.py --predictions /tmp/preds.json
```

Generate the current manual baseline:

```bash
cd /home/dino/context-farm-repo
python3 scripts/generate_manual_baseline_predictions.py
python3 scripts/eval_extraction.py \
  --predictions examples/service-dispatch-extraction-predictions.manual-baseline.json
```

Run the live staged benchmark against the configured OpenAI-compatible endpoint:

```bash
cd /home/dino/context-farm-repo
scripts/run_extraction_benchmark.sh
```

By default the runner uses:

- `BASE_URL=http://100.120.50.35:8010/v1`
- `MODEL=local`
- `PIPELINE_NAME=llm-staged-live`
- `OUTPUT_DIR=artifacts/extraction-benchmarks/latest`

## Metrics

The evaluator reports:

- prediction coverage
- type accuracy
- exact and graded title match
- strict and graded statement coverage
- review-priority accuracy

Type and review priority are categorical, so they remain exact-match metrics.

Title and canonical statement are generated text, so exact-string matching is not a fair primary score. The evaluator therefore reports both:

- strict metrics, to preserve the hard view
- graded overlap/recall metrics, to judge extraction quality without requiring identical wording

This is still intentionally simple. It is the first honest extraction scoreboard, not the final one.

## Current Benchmarks

There are now two benchmark families:

- `manual-object-overlap-baseline`: generated by matching gold examples against the manual demo object set
- `llm-staged-extractor-v0` and `llm-staged-extractor-v1`: cold-from-text staged extraction, with v1 adding disambiguation rules

The manual baseline is still useful as a comparison point, but it is not a true extraction benchmark. It can score highly by selecting from objects that already encode the answer key.

### Manual Baseline

Pipeline: `manual-object-overlap-baseline`

- coverage: `100.0%`
- type accuracy: `81.2%`
- priority accuracy: `87.5%`
- title exact: `68.8%`
- title graded: `87.5%`
- statement strict: `62.5%`
- statement graded: `62.5%`

Known misses are mostly artifact-shape problems:

- `sd-010`, `sd-013`, and `sd-016`: the baseline points at the right object family but misses the span-specific phrasing badly enough to fail statement coverage
- `sd-011`: the manual object set does not contain a dedicated documentation constraint object
- `sd-014`: the manual object set has no matching manager decision about tightening VIP waivers
- `sd-015`: the manual baseline resolves a procedural intake step to the broader emergency-dispatch object and loses the pre-approval sequence

### Staged LLM Baseline v0

Pipeline: `llm-staged-extractor-v0`

- coverage: `100.0%`
- type accuracy: `56.2%`
- priority accuracy: `81.2%`
- title graded: `56.2%`
- statement graded: `93.8%`

Known misses:

- `sd-002` and `sd-016`: the model over-indexes on carve-out language and promotes primary constraints into exceptions
- `sd-005`, `sd-006`, and `sd-010`: the model still collapses operator action flows into constraints or exceptions
- `sd-009` and `sd-014`: the model treats manager policy decisions as plain constraints

### Staged LLM Baseline v1

Pipeline: `llm-staged-extractor-v1`

- coverage: `100.0%`
- type accuracy: `75.0%`
- priority accuracy: `87.5%`
- title graded: `62.5%`
- statement graded: `87.5%`

Known remaining mismatches:

- `sd-006`: `Procedure` vs `Constraint`
- `sd-009`: `Decision` vs `Constraint`
- `sd-010`: statement is correct, but the model still classifies the notify/escalate step as a `Constraint`
- `sd-013`: type is correct, but the generated statement drops the "before same-day service can be promised" enforcement detail
- `sd-014`: statement is correct, but the model still treats an "effective immediately" policy change as a `Constraint`

## Why This Matters

Context Farm already has:
- a manual demo domain
- a structured object schema
- a SQLite store
- retrieval and briefing prototypes

What it does not yet have is a disciplined way to say whether the extraction loop is getting better.

This artifact is the first answer to that problem.
